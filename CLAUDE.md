# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Three independent single-file Node.js tools for Claude Code, no dependencies beyond Node's built-in modules (`fs`, `path`, `os`, `readline`, `child_process`, `url`, `https`) and no build step, package manager, or test suite — all are meant to be run directly with `node`.

- **`statusline.js`**: implements both a main status line and a subagent status line.
- **`loopctl.js`**: an opt-in, per-project auto-continue loop built on Claude Code's `Stop` hook. `statusline.js` reads (but never writes) the state file this tool manages, to show a loop badge.
- **`pricing-updater.js`**: fetches live model pricing from the community-maintained litellm price table and writes/merges `~/.claude/pricing.json` (or a chosen `--out` path — the maintainer uses `--out pricing.json` to refresh the repo's shipped copy), which `statusline.js` reads to self-compute session cost. Only this tool touches the network; the status line itself never does.

## Commands

There is no build/lint/test tooling in this repo. Useful commands:

- Run the status line manually against sample main-session input: `echo '{}' | node statusline.js`
- Run it against mock subagent-row input (must include a `tasks` array to trigger that code path): `echo '{"tasks":[{"id":"t1","name":"Explore","tokenCount":100,"startTime":"2026-01-01T00:00:00Z"}]}' | node statusline.js`
- Install/update the status line as both the active `statusLine` and `subagentStatusLine`: `node statusline.js --install` (copies itself to `~/.claude/statusline.js` and updates both fields in `~/.claude/settings.json`, prompting independently before overwriting either existing file/setting) — same thing via `install.bat` on Windows or `npm run install-statusline`
- Install loopctl as a global tool + register its `Stop` hook: `node loopctl.js --install` (appends to `settings.json`'s `hooks.Stop` array rather than replacing it, so it coexists with other Stop hooks) — or `npm run install-loopctl`
- Toggle the loop in the current project: `node loopctl.js on --max 8 --push`, `node loopctl.js off`, `node loopctl.js status`
- List built-in prompt presets: `node loopctl.js presets`
- Simulate a Stop hook invocation manually: `echo '{"cwd":"'"$(pwd)"'"}' | node loopctl.js --hook`
- Quick syntax check for both files: `npm run check` (runs `node --check` on both)

## Architecture

The script runs in two distinct top-level modes based on `process.argv`:

1. **Install mode** (`--install` flag): `installStatusLine()` copies the running script to `~/.claude/statusline.js` and wires up `~/.claude/settings.json` so Claude Code invokes it as both `statusLine` and `subagentStatusLine` (same file, same command — the script tells the two inputs apart at runtime, see below). It diffs file content / existing settings before touching anything and prompts the user via `promptYesNo()` independently for the script copy, the `statusLine` setting, and the `subagentStatusLine` setting.

2. **Render mode** (default, no args): Claude Code invokes the script as a subprocess and pipes a JSON payload on stdin. The script reads all of stdin, parses it once `process.stdin` emits `'end'`, and writes to stdout. Within render mode there are two payload shapes, distinguished by whether `data.tasks` is an array:
   - **Subagent rows** (`subagentStatusLine`): payload is `{ columns, tasks: [...] }`, one entry per visible subagent row (`id`, `name`, `type`, `status`, `tokenCount`, `startTime`, `tokenSamples`, `contextWindowSize`, `effort`, ...). Handled by `renderSubagentStatusLine()`, which must emit one NDJSON line per task as `{"id": "<task id>", "content": "<row>"}` — omitting a task's `id` keeps Claude Code's default row rendering, and an empty `content` hides it.
   - **Main session** (`statusLine`): payload has session info, `cwd`, `transcript_path`, `cost`, `context_window`, `rate_limits`, etc. This is the original/default path.

Key implementation detail: **output-token/timing data for the main status line is not present in the stdin JSON.** Claude Code's stdin payload only has coarse session cost/duration. To compute output tokens-per-second (TPS), the script re-parses the JSONL transcript file at `data.transcript_path` line-by-line, pairing each `assistant` message's `usage.output_tokens` with the timestamp of the preceding `user` message (prompt or tool_result) to derive elapsed generation time. It also accumulates session-wide input/output token totals from every assistant turn in the transcript (excluding cache-read tokens from the input sum, since those repeat every turn and would inflate the total).

For subagent rows, there is no transcript to fall back on — the per-task rate comes from `tokenSamples` (shape undocumented upstream; `estimateRecentRate()` defensively probes for `{timestamp, tokens}`-like entries and returns `null` if the shape doesn't match) or, failing that, `tokenCount / (now - startTime)` as a coarse fallback. This is deliberately labeled `tok/s` rather than `TPS` in the output since it may reflect running context usage rather than pure generation throughput.

Other notable behaviors:
- `findGitDir()` / `getGitBranch()` resolve the current git branch by reading `.git/HEAD` directly (handling both real `.git` dirs and `gitdir:` pointer files used by worktrees/submodules) for speed, falling back to spawning `git branch --show-current` only if that fails.
- `buildBranchUrl()` turns `workspace.repo.{host,owner,name}` (already parsed by Claude Code from the `origin` remote — no extra `git remote` call needed) plus the resolved branch name into a clickable "browse this branch" URL. Each host has its own path shape (GitHub/GitLab use `/tree/`-ish paths, Bitbucket uses `/src/`); unrecognized hosts (self-hosted Git, etc.) return `null` and the branch renders as plain text rather than a guessed-wrong link. A detached-HEAD short hash (from `getGitBranch`'s fallback) also returns `null` since it isn't a branch page. When a URL is available, the branch segment is wrapped via `osc8()` (the generic OSC-8 helper; `osc8FileLink()` is the cwd-specific wrapper around it).
- `readLoopState()` is a read-only peek at `<currentDir>/.claude/loop-state.json`, the state file `loopctl.js` owns (see below). `statusline.js` never writes to it — toggling the loop is exclusively loopctl's job. When `enabled` is true, a `🔁loop:<round>/<max>` badge appends to the first line; absent otherwise.
- Main status line output is two lines: line 1 is identity/workspace (model, OSC-8 clickable `file://` link to the cwd, git branch — itself an OSC-8 link to the branch's browse page when `buildBranchUrl()` resolves one, PR number/review state, session name, loop badge, context-window usage %); line 2 is activity/cost/mode (TPS, output tokens, cache tokens, session totals, lines added/removed, estimated cost, duration, 5h/7d rate-limit %, fast mode, extended thinking, reasoning effort, vim mode) — trailing fields are only emitted when present. Colors are applied via raw ANSI escape codes (`ANSI` object), not a library; `colorForPercentage()` is the shared red/yellow/green threshold helper used by both context % and rate-limit %.
- `cost.total_cost_usd` is a client-side estimate documented as potentially diverging from the actual bill, and it is priced at Anthropic's rates even when you route to a third-party model (e.g. MiniMax via `ANTHROPIC_BASE_URL`), so the script does **not** display it verbatim. Instead it self-computes cost from the transcript token counts (split into fresh input / output / cache-read / cache-creation, accumulated session-wide in the parse loop) times a public, editable pricing table that lives in **`pricing.json` next to the script** — one file, the single source of truth (no hardcoded `PRICING` constant anymore; the table was externalized so the repo ships and maintains exactly one pricing file). `loadPricing()` reads `path.join(__dirname, 'pricing.json')` best-effort (missing/invalid → `{}`, in which case cost falls back to the client estimate); `resolvePricing(model, table)` matches by exact name then longest case-insensitive substring; `computeCost()` is the rate×tokens formula. The self-computed figure is shown as `~cost:`; when the current model has no pricing entry (and only then) it falls back to `total_cost_usd` shown as `~cost?:` to flag that it's the untrusted client estimate. Both are estimates — hence the `~` prefix — don't "fix" this by dropping the prefix. `--install` seeds `~/.claude/pricing.json` from the repo's bundled `pricing.json` **only if the target doesn't exist** (never overwrites — the user may have refreshed it), so cost works out of the box. `pricing.json` is maintained by `pricing-updater.js`, which fetches live rates from the litellm price table (sourced from each provider's official pricing page — no official JSON API exists) and merges them into `~/.claude/pricing.json` on demand — **manual run only, no scheduled task**, by design (silently rewriting rate data in the background is risky). The maintainer refreshes the repo's shipped copy with `--out pricing.json --overwrite` and commits; users refresh their own with a bare `pricing-updater` run. `pricing-updater.js`'s `defaultKeys()` extracts the canonical chat model of every mainstream provider (Anthropic, OpenAI, Google/Gemini, MiniMax, DeepSeek, Meta/Llama, Mistral, xAI/Grok, Cohere, Qwen/Dashscope) with all versions: it filters `mode === 'chat'`, keeps only `MAINSTREAM_PROVIDERS` entries, and `isCanonicalKey()` rejects hoster/regional duplicates (bedrock/vertex/openrouter/azure/…, 2+ slashes) and `ft:` fine-tunes. Each model dedupes to one friendly key (substring after the last `/`), preferring bare no-slash keys so a collision like `deepseek-chat` (bare) vs `deepseek/deepseek-chat` resolves to the bare one. Don't relax the canonical filter without re-reading why — without it every model explodes into dozens of regional duplicates that collide on the friendly key.
- At the very end, there's a best-effort, opt-in integration that broadcasts session activity to an "AutoClaude" sensor. It is gated on the `AUTOCLAUDE_BROADCAST` env var, which must point at a `broadcast.js` module exporting `buildFrame`/`sendFrame`. When unset (or the path is missing), the block is a no-op. This is wrapped in try/catch and must never block or break the status line — treat it as optional/environment-specific, not a project dependency to package or generalize. This only runs in the main-session path, not the subagent-row path.

### Auto-publish to npm

`.github/workflows/publish.yml` runs on every push to `main`. It compares the version in `package.json` to the latest published version on npm; only when they differ does it: (1) publish to `@tangjianfang/claudecode-statusline`, (2) publish to `@tangjianfang/cc-statusline` (after swapping `name` in the ephemeral checkout), (3) push the matching `vX.Y.Z` git tag, (4) create the GitHub Release via `softprops/action-gh-release` with auto-generated notes. `--ignore-scripts` is used so the publish step doesn't recursively trigger the repo's own `postinstall`. Required secret: `NPM_TOKEN` (granular, bypass-2fa, publish scope on both packages). The release flow is documented in README under "Releasing a new version (maintainer)".

### loopctl.js

`loopctl.js` turns Claude Code's `Stop` hook into an opt-in auto-continue loop, per project. State lives at `<project>/.claude/loop-state.json` (project-local, not `~/.claude/`) so the loop is off everywhere by default and has to be turned on explicitly per project — this is deliberate per the user's requirement that it never affect projects the user hasn't opted into.

Modes, dispatched from `process.argv` the same way `statusline.js` dispatches `--install`:
- `on [--max N] [--push] [--no-push] [--preset <name>] [--prompt "..."]` / `off` / `status` / `presets`: CLI commands (`cmdOn`/`cmdOff`/`cmdStatus`/`cmdPresets`) that read-modify-write the state file for `process.cwd()`.
- `--install`: copies itself to `~/.claude/loopctl.js` and appends an entry to `settings.json`'s `hooks.Stop` array (does not overwrite existing Stop hooks — Stop supports multiple hooks running side by side).
- `--hook`: the actual Stop-hook entry point (`runHook()`). Reads the hook's stdin JSON, looks up `data.cwd` in the state file, and either emits nothing (let Claude stop normally — the correct behavior whenever the loop is off, round cap reached, or `data.stop_hook_active === true`) or emits `{"decision":"block","reason":"<prompt> (auto-loop round N/max)"}` to make Claude continue. `decision:"block"` plus a `reason` string is the documented Stop-hook mechanism for injecting the next instruction; there's no `stdout` schema beyond that.

The `PRESETS` object (top of the file) holds named prompt templates (`next-step`, `find-bugs`, `improve`, `tests`, `docs`, `refactor`) selectable via `--preset <name>` instead of typing a full `--prompt` each time; `resolvePreset()` throws on an unknown name and `parseOnArgs()`/`cmdOn()` propagate that as a CLI error rather than silently falling back to the default. `--prompt` parsed after `--preset` on the command line overrides it (last-write-wins in `parseOnArgs`'s left-to-right loop) — this lets a preset serve as a starting point for a one-off tweak. Add new goals to `PRESETS` rather than telling users to hand-write long `--prompt` strings.

Key constraints baked into the design, don't relax without re-reading why:
- **Round cap is intentionally checked against Claude Code's own protection**: Claude Code force-releases a Stop hook after it blocks 8 times in a row without the hook reporting different state (`stop_hook_active` guard exists for exactly this). `DEFAULT_MAX_ROUNDS = 8` matches that ceiling; `cmdOn()` warns (doesn't block) when `--max` exceeds it, since raising the effective ceiling requires the user to also set `CLAUDE_CODE_STOP_HOOK_BLOCK_CAP` — this script has no way to change that env var for the user's session.
- **`--push` runs `git add -A && git commit && git push` via `spawnSync` with an argv array**, never a shell string — the commit message is built from user-controlled `state.prompt`, so string-interpolating it into a shell command would be a command-injection risk. Push failures (no remote, network, etc.) are caught and folded into the `reason` text rather than thrown, so a broken push never blocks the loop or crashes the hook — but the commit itself still lands locally.
- **Push targets whatever branch is currently checked out, including `main`** — there's no dedicated-branch safety net. This was an explicit user choice, not an oversight; don't add branch restrictions without asking first.
- `stateFilePath()`/`readState()`/`writeState()` are the only read/write surface for the state file; `statusline.js`'s `readLoopState()` duplicates the read logic (deliberately, to keep the two tools independent) rather than importing from here.
